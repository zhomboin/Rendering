## Why this matters

只要开始讨论 MySQL 的并发读写，MVCC 就几乎绕不过去。很多人知道它和“快照读”“undo log”“Read View”有关，但一到面试、排查线上读写结果不一致，或者解释为什么 `READ COMMITTED` 和 `REPEATABLE READ` 行为不同的时候，概念就很容易散掉。

这篇文章的目标不是把名词再列一遍，而是把几个关键部件真正串起来：一行数据为什么会有多个版本，查询时到底怎么决定“看见哪个版本”，以及这个过程为什么刚好支撑了 MySQL 的隔离级别设计。

## What is happening

可以把 InnoDB 的 MVCC 理解成一句话：

> 写操作持续产生“新版本”，读操作根据自己的视角挑选“可见版本”。

这样做的直接收益是，普通查询不需要和写操作长期互相等待。写事务可以继续修改当前版本，读事务则沿着版本链回溯到自己应该看到的历史快照，从而避免大量读写冲突。

这里的关键不在“多版本”四个字，而在三件事同时成立：

1. 行记录里保存了版本相关的隐藏信息。
2. 旧版本没有被立刻抹掉，而是通过 undo log 串起来。
3. 每次读操作都会带着自己的 Read View，按照规则判断某个版本是否可见。

## Walkthrough

### 先抓住三个底层对象

在 InnoDB 里，一行记录除了业务字段之外，还会有几列对 MVCC 很重要的隐藏信息：

- `DB_TRX_ID`：最后一次修改这行记录的事务 ID。
- `DB_ROLL_PTR`：指向 undo log 的指针，用来回到更早版本。
- `DB_ROW_ID`：行 ID，通常只在没有显式主键时承担内部标识作用。

其中真正决定版本链条的是前两项。更新发生时，当前值不会“凭空消失”，而是先把旧值写进 undo log，再让最新版本挂到表里。这样一来，查询既能拿到最新记录，也能顺着 `DB_ROLL_PTR` 一层层回溯。

### Read View 到底记录了什么

一致性读并不会盲目相信“现在表里最新的那一行”，它会先拿到一个 Read View。你可以把 Read View 看成事务当下的并发现场快照，里面最值得记住的是：

- `creator_trx_id`：当前读视图属于哪个事务。
- `trx_ids`：创建 Read View 时，系统里还没提交的活跃事务列表。
- `low_limit_id`：创建 Read View 时，系统将要分配的下一个事务 ID。

只靠这几项信息，MySQL 就能判断“某个版本是不是对当前事务可见”。

### 一次查询如何判断当前版本可不可见

可以把可见性判断粗略记成下面这条思路：

1. 先看当前记录版本的 `DB_TRX_ID`。
2. 如果它属于当前事务自己创建的数据，通常可见。
3. 如果它出现在 `trx_ids` 里，说明这个版本来自仍未提交的事务，对当前读不可见。
4. 如果它比 `low_limit_id` 还新，说明这个版本是在当前 Read View 创建之后才出现的，也不可见。
5. 当前版本不可见时，就顺着 `DB_ROLL_PTR` 去 undo log 找更老的版本，直到找到第一个可见版本为止。

也就是说，MVCC 从来不是“只看一份神秘快照”，而是拿着一套可见性规则，在版本链上向后回溯。

### 用三笔事务把过程串起来

下面这组 SQL 很适合把整个过程串起来：

```sql
create table t1 (
  id int primary key,
  c2 int
);

insert into t1 values (1, 100);
```

然后让三个事务交错执行：

```sql
-- 事务 1
start transaction;
update t1 set c2 = 200 where id = 1;

-- 事务 2 第一次查询
start transaction;
select * from t1 where id = 1;

-- 事务 3
start transaction;
update t1 set c2 = 300 where id = 1;

-- 事务 2 第二次查询
select * from t1 where id = 1;

-- 事务 1 提交后，事务 2 第三次查询
select * from t1 where id = 1;
commit;
```

假设事务 ID 依次是：事务 1 为 `120`，事务 2 为 `130`，事务 3 为 `140`。

如果事务 2 运行在 `READ COMMITTED` 下，那么它每次查询都会重新创建 Read View：

- 第一次查询时，事务 1 还没提交，因此最新版本的 `DB_TRX_ID = 120` 落在活跃事务里，不可见，只能回滚到旧值 `100`。
- 第二次查询时，事务 3 又写出了一个新版本，`DB_TRX_ID = 140` 同样不可见；继续往回追，还会遇到事务 1 产生的版本 `120`，依然不可见，所以最终仍看到 `100`。
- 第三次查询时，事务 1 已提交，事务 3 仍未提交，于是版本 `140` 还是不可见，但版本 `120` 已经变成可见版本，因此这次结果变成 `200`。

这就是为什么 `READ COMMITTED` 可能在同一事务里读到不同结果：不是 MySQL 在“随机返回数据”，而是每次查询拿到的 Read View 都在更新。

### 读已提交和可重复读真正差在哪

两者都依赖 MVCC，但差异不在“有没有快照”，而在“快照什么时候生成”。

- `READ COMMITTED`：每次一致性读都会生成新的 Read View。
- `REPEATABLE READ`：事务第一次一致性读生成 Read View，后续继续复用这一份视图。

所以 `REPEATABLE READ` 的稳定性来自“视角固定”，而不是“不允许别人修改数据”。别人依然可以写，只是你在当前事务里继续做普通快照读时，看见的还是第一次建立视角时能看到的版本。

## Practical checks

如果你想判断某个查询结果是不是 MVCC 的典型表现，可以按下面的顺序做检查：

1. 先确认当前隔离级别。

```sql
select @@session.transaction_isolation;
```

2. 再判断这是不是普通一致性读，而不是 `for update` 这类锁定读。
3. 如果结果和预期不一致，回到“Read View 何时创建”这个问题，而不是先怀疑 MySQL 读错了数据。
4. 在分析更新链条时，优先从“最新版本为什么不可见”入手，再顺着 undo log 的逻辑往回推。

一个很实用的心智模型是：

- 锁解决“谁能改”。
- MVCC 解决“谁能看到哪个版本”。

把这两件事分开，很多并发问题会清楚很多。

## Closing notes

MVCC 最容易让人困惑的地方，是它看起来像一个抽象概念，但真正落地时又完全依赖很具体的版本链和可见性判断。只要把隐藏列、undo log、Read View 和隔离级别四件事放在同一张图里看，它其实并不神秘。

如果你后面还要继续往下学，我建议下一步直接把 MVCC 和隔离级别实验放在一起做。单独理解 MVCC 很容易停留在概念层，和 `READ COMMITTED`、`REPEATABLE READ` 的实际行为对照起来，才会真正形成稳定直觉。
