# 网站正式上线 Checklist + 发文 SOP

这份文档面向当前阶段的 `Rendering` 个人博客。

适用范围：

- 当前站点定位为个人博客，而不是 CMS
- 站点级国际化已支持中英文，默认中文
- 文章正文当前不做双语翻译
- 内容发布方式为 `Git + MDX + 自动构建发布`

## 1. 文档目标

这份文档解决两件事：

- 网站正式上线前，我们需要确认哪些条件已经满足
- 网站上线后，我们如何稳定、低成本地持续发文

## 2. 上线前必须满足的条件

以下项目建议视为正式上线前的 `Go / No-Go` 条件。

### 2.1 品牌与站点信息

- [ ] 确认正式域名，并与站点 SEO 配置保持一致
- [ ] 确认站点名称、作者名称、站点描述为真实可公开信息
- [ ] 确认 About 页内容已完成，不再保留占位文案
- [ ] 确认 favicon、分享图、站点描述符合当前品牌方向

当前需要重点核对的文件：

- `lib/seo.js`
- `lib/messages/zh.js`
- `lib/messages/en.js`
- `app/manifest.ts`

### 2.2 部署与自动发布

- [ ] 选定正式部署平台
- [ ] 正式域名已正确解析到部署平台
- [ ] `main` 分支推送后可自动触发构建与发布
- [ ] GitHub Actions 的 CI 已稳定转绿
- [ ] 构建产物中的 `feed.xml`、`sitemap.xml`、`robots.txt`、`manifest.webmanifest` 可正常访问

当前项目的核心构建链路：

- `npm run build`
- 站点构建完成后自动生成搜索索引
- SEO 与 Feed 路由由项目本身生成，无需手工维护静态文件

### 2.3 内容与公开状态

- [ ] 至少准备一批愿意公开展示的正式文章
- [ ] 确认不应公开的文章仍然保持 `draft: true`
- [ ] 每篇公开文章都具备完整标题、描述、发布日期、标签
- [ ] 标签结构已能支持首页、归档页、标签页的浏览体验

### 2.4 质量与体验

- [ ] 桌面端首页、博客列表、标签页、文章页、About 页完整走查一遍
- [ ] 移动端首页、文章页、标签页完整走查一遍
- [ ] 明暗主题切换正常
- [ ] 中文默认路由与英文镜像路由正常
- [ ] 搜索弹层、标签筛选、标签归档、文章阅读进度正常
- [ ] 文章图文混排、图库、lightbox 正常

### 2.5 工程校验

正式上线前，至少需要完成一次完整校验：

```bash
npm run check
```

它会覆盖当前项目里的关键检查：

- 内容校验
- 单元测试
- 类型检查
- 生产构建

## 3. 正式上线 Checklist

建议按下面的顺序执行。

### 3.1 上线前一天

- [ ] 核对 `lib/seo.js` 中的正式域名、作者名、站点信息

- [ ] 清点公开文章列表，确认哪些文章要首批上线
- [ ] 再次运行 `npm run check`

### 3.2 上线当天

- [ ] 将最终代码合入并推送到 `main`

- [ ] 用英文路由检查 `/en`、`/en/blog`、`/en/about`

## 4. 当前网站的发布方式

当前网站不是 CMS，也没有后台发布入口。

当前标准发文方式是：

`本地编写 MDX -> 本地校验 -> 提交 Git -> push 到 main -> 自动构建发布`

也就是说，后续发布文章不需要登录后台，而是直接在仓库中新增或修改 `content/posts/*.mdx` 文件。

## 5. 发文 SOP

### 5.1 新建文章

推荐使用项目内置脚本创建新文章：

```bash
npm run create-post -- <slug> [tag1,tag2]
```

示例：

```bash
npm run create-post -- mysql-locking-read mysql,innodb
```

脚本会自动：

- 规范化 `slug`
- 生成文章文件
- 生成基础 frontmatter
- 默认将文章设为 `draft: true`

生成位置：

- `content/posts/<slug>.mdx`

### 5.2 填写 frontmatter

每篇文章至少需要以下字段：

```yaml
---
title: "文章标题"
description: "一段用于列表页和 SEO 的摘要"
publishedAt: "2026-03-22"
tags:
  - mysql
  - innodb
draft: true
---
```

规则说明：

- `title` 必填
- `description` 必填
- `publishedAt` 必须是 `YYYY-MM-DD`
- `tags` 对于公开文章必须至少有一个
- `draft: true` 表示暂不公开
- 准备上线时再改成 `draft: false`

### 5.3 编写正文

正文直接使用 `MDX`。

当前建议：

- 技术文章优先保持清晰结构
- 优先使用真实命令、真实配置、真实排错过程
- 长文中适当使用二级、三级标题帮助导航
- 如有图片，统一放在 `public/` 下管理
- 正文无需双语；当前国际化只覆盖站点壳层

如果文章包含图片或图组，建议沿用现有图文组件能力：

- `figure`
- `caption`
- `gallery`
- 点击放大预览

### 5.4 本地预览

写作完成后，本地启动开发环境预览：

```bash
npm run dev
```

重点检查：

- 列表页中的标题和摘要是否顺畅
- 文章页标题区是否正常
- 图文混排是否稳定
- 标签是否合适
- 目录、上一篇/下一篇、阅读进度是否正常

### 5.5 本地校验

发文前统一执行：

```bash
npm run check
```

如果只想快速检查内容，可先运行：

```bash
npm run lint
```

当前这一步主要会校验：

- frontmatter 是否完整
- `publishedAt` 格式是否正确
- 标签是否为空
- 站点是否可正常构建

### 5.6 发布文章

当文章确认可以公开时：

1. 将 frontmatter 中的 `draft` 改为 `false`
2. 再次运行 `npm run check`
3. 提交代码
4. push 到 `main`
5. 等待部署平台自动发布

发布完成后，文章会自动进入：

- 首页文章区域
- 博客归档页
- 标签页与标签归档页
- `sitemap.xml`
- `feed.xml`
- 搜索索引

### 5.7 发布后检查

每篇新文章发布后，建议至少做一次快速检查：

- [ ] 文章详情页可正常打开
- [ ] 文章标题与摘要显示正确
- [ ] 标签跳转正常
- [ ] 分享图正常
- [ ] Feed 中已出现新文章
- [ ] 搜索可以搜到该文章

## 6. 迁移旧文 SOP

当前已经确认的策略：

- 可从已有原创内容渠道迁移文章
- 本站作为首发站使用
- 迁移后的文章默认保留原文内容表达
- 不要求再经过统一模板化改写

迁移旧文时建议遵循下面的最小流程：

1. 先确定目标 `slug`
2. 新建对应 `.mdx` 文件
3. 补齐 frontmatter
4. 将正文迁入，并清理平台噪声内容
5. 本地检查排版、代码块、图片、标题层级
6. 保持 `draft: true` 先预览
7. 确认无误后再改为 `draft: false`

推荐额外检查：

- 是否保留了不适合本站的外链引导
- 是否存在平台专属提示语
- 是否需要补图片或代码块格式
- 是否需要统一标签命名

## 7. 日常发文建议节奏

对于当前阶段的个人博客，推荐采用下面的节奏：

- 新文章优先保证质量，不追求后台式高频更新
- 每次发文尽量带着明确标签归类
- 每次发布后顺手检查 Feed、搜索和标签归档
- 每累计 3-5 篇新文章，就回头整理一次标签结构

## 8. 当前阶段不包含的发文方式

以下内容不在当前阶段范围内：

- 后台可视化发文
- 登录后在线编辑
- CMS 审核流
- 多作者协同发布流
- 数据库驱动的内容管理

也就是说，当前最稳定、最适合本项目的内容发布方式，仍然是：

`本地写作 + Git 提交 + 自动部署`

## 9. 一句话执行版

如果只保留最短流程，当前发文流程就是：

1. `npm run create-post -- <slug> [tags]`
2. 写 frontmatter 和正文
3. 本地 `npm run dev` 预览
4. 运行 `npm run check`
5. 将 `draft` 改为 `false`
6. 提交并 push 到 `main`
7. 发布后检查文章、Feed、搜索、标签页
