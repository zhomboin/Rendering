# Rendering 开发计划

日期：2026-03-19
项目：Rendering
文档状态：Active
文档角色：当前个人博客阶段的唯一中文总开发计划

## 1. 计划目标

将 `Rendering` 交付为一个完整的个人技术博客，具备高质量公开阅读体验、本地 MDX 内容工作流、静态搜索与 SEO 基础能力；同时保持代码结构对未来 CMS 扩展友好，但当前阶段不实际实现 CMS 功能。

## 2. 计划前提

本计划默认以下前提成立：

- 当前产品是个人博客
- 公开阅读体验是首要交付目标
- 本地 MDX 是当前权威内容工作流
- CMS 相关能力延后到未来阶段文档中定义

本计划应作为当前阶段所有后续工作的实施依据。

## 3. 当前阶段边界

### 本计划涵盖

- 公开路由和读者侧 UI
- 本地内容架构
- 视觉系统与交互质量
- 搜索与内容发现
- SEO 基础能力
- 质量、可维护性与部署准备

### 本计划不涵盖

- 登录与鉴权
- 后台管理面板
- 基于数据库的内容编辑
- 多作者发布
- 角色与权限模型
- 后端媒体管理

## 4. 当前状态概览

仓库已经具备当前阶段的大部分基础能力：

- 公开页面
- MDX 内容仓库层
- Pagefind 搜索集成
- 基础 metadata

因此，这份计划同时承担两个作用：

- 确立当前个人博客阶段的唯一执行目标
- 为后续打磨和增量工作提供清晰优先级

## 5. 总体交付策略

当前个人博客阶段建议按六条工作流推进：

1. 产品对齐与文档基线
2. 公开博客体验完善
3. 内容系统与编辑工作流
4. 搜索与 SEO
5. 质量、性能与部署准备
6. 未来 CMS 扩展准备

## 6. 工作流 1：产品对齐与文档基线

### 目标

把“个人博客优先”的范围彻底写清楚，避免设计和开发继续向“提前做 CMS”漂移。

### 任务

- 在 `docs/PRD.md` 中维护唯一英文主 PRD
- 在 `docs/development-plan.md` 中维护唯一英文主开发计划
- 为当前阶段补充对应中文文档，降低沟通成本
- 将 `docs/superpowers/specs/` 和 `docs/superpowers/plans/` 视为历史迭代资料与未来扩展槽位，而不是当前范围的权威来源
- 以后新增需求时，先对照当前 PRD 判断是否属于个人博客阶段

### 完成标准

- 当前阶段存在单一主 PRD 和单一主开发计划
- 中英文文档都能清晰表达当前范围
- 团队能明确区分“当前博客范围”和“未来 CMS 范围”

## 7. 工作流 2：公开博客体验完善

### 目标

持续打磨读者端产品，使其像一个完成度高的个人出版站点，而不是一个未完工的后台前壳。

### 核心范围

- 首页叙事和层级
- 归档页扫描效率
- 文章页阅读舒适度
- 标签驱动浏览
- About 页的人设与项目定位
- 响应式体验

### 关键文件

- `app/layout.tsx`
- `app/page.tsx`
- `app/blog/page.tsx`
- `app/blog/[slug]/page.tsx`
- `app/tags/page.tsx`
- `app/about/page.tsx`
- `app/globals.css`
- `components/`

### 任务

- 把首页持续打磨成“编辑入口页”，而不是产品营销页
- 保持归档页比首页更高密度、更利于快速浏览
- 把文章页可读性当作整个产品的最高优先级
- 让标签页成为真实导航面，而不是被动元数据列表
- 让 About 页更像作者与项目说明，而不是简历页
- 验证小屏幕下的行宽、间距、层级与 focus 可见性

### 完成标准

- 所有公开页面风格统一、功能清晰
- 站点整体感受是个人技术博客，而不是 dashboard 或 CMS 外壳
- 文章页持续是产品中最强的体验

## 8. 工作流 3：内容系统与编辑工作流

### 目标

让发布流程保持简单、稳定、对作者友好，同时为未来扩展留下余地。

### 核心范围

- MDX 写作
- frontmatter 契约
- 内容仓库层稳定性
- 草稿处理
- 标签体系一致性

### 关键文件

- `content/posts/*.mdx`
- `lib/content.js`
- `lib/mdx.js`
- `lib/content.test.js`
- `components/mdx-components.tsx`

### 任务

- 保持本地 MDX 为当前主写作流程
- 维持 title、description、publishedAt、tags、draft 的稳定 frontmatter 契约
- 随着内容增加，保证标签聚合逻辑仍然可靠
- 保证页面仅通过 repository helpers 获取公开内容
- 在不复杂化作者体验的前提下，持续优化 MDX 文章样式
- 任何内容层行为变更都补充测试

### 完成标准

- 新文章发布流程低摩擦
- 草稿内容不会泄漏到公开页面
- 内容逻辑易于理解、维护和测试

## 9. 工作流 4：搜索与 SEO

### 目标

在不引入后台复杂度的前提下，让已发布内容更容易被发现和传播。

### 核心范围

- Pagefind 索引
- 搜索体验
- 路由 metadata
- 文章 metadata
- 后续 SEO 增强项

### 关键文件

- `components/search-modal.tsx`
- `components/search-trigger.tsx`
- `components/site-header.tsx`
- `scripts/build-search-index.mjs`
- `app/layout.tsx`
- `app/page.tsx`
- `app/blog/page.tsx`
- `app/blog/[slug]/page.tsx`
- `app/tags/page.tsx`
- `app/about/page.tsx`

### 任务

- 保持搜索弹层和整体编辑型视觉语言一致
- 确保快捷键、focus 恢复和键盘交互稳定
- 保持文章 metadata 由 frontmatter 驱动
- 维持 canonical URL 规则统一
- 在当前阶段后续视价值补齐以下 SEO 能力：
- `sitemap.xml`
- `robots.txt`
- JSON-LD
- 更丰富的 OG 支持

### 完成标准

- 搜索对现有内容库足够有效
- 各公开路由 metadata 正确
- 站点具备继续扩展 SEO 的干净基础

## 10. 工作流 5：质量、性能与部署准备

### 目标

让个人博客在稳定性、可维护性和上线准备上都足够可靠。

### 核心范围

- 构建稳定性
- 内容层测试
- 响应式 QA
- 可访问性检查
- 视觉一致性

### 任务

- 持续保持内容仓库层测试通过
- 持续保持生产构建和搜索索引构建成功
- 验证文章页在移动端和桌面端都保持良好阅读体验
- 保持清晰 focus 状态和 reduced-motion 支持
- 避免使站点变得通用化或噪声化的视觉回退
- 保持部署路径简单、静态友好

### 完成标准

- `npm.cmd test` 通过
- `npm.cmd run build` 通过
- 公开路由具备生产可用性
- 当前阶段可以作为个人博客稳定部署

## 11. 工作流 6：未来 CMS 扩展准备

### 目标

只做合理预留，不在当前阶段实际构建 CMS。

### 当前允许的准备工作

- 将内容访问统一封装在 repository helpers 后
- 避免页面直接耦合文件解析细节
- 为未来替换内容源保留结构空间
- 让 UI 组件在未来后台阶段仍可复用

### 当前不允许做的事

- 为“也许以后会用”而提前做登录流程
- 在当前没有明确需求时加入数据库依赖
- 引入并不真正需要的 admin 路由
- 按未来 dashboard 的逻辑来反向塑造当前公开博客

### 未来文档策略

当项目准备进入 CMS 扩展阶段时，再在以下目录建立新的分阶段文档：

- `docs/superpowers/specs/`
- `docs/superpowers/plans/`

可能的未来阶段包括：

- 作者鉴权
- 后台内容管理
- 数据库化发布流程
- 媒体资源工作流
- 评论与审核系统

## 12. 推荐优先级顺序

当前个人博客阶段剩余工作的推荐顺序为：

1. 先稳定 PRD 与计划基线
2. 持续打磨公开阅读与浏览体验
3. 强化内容质量与作者工作流纪律
4. 在有明确价值时补强 SEO
5. 继续收紧 QA 和部署准备
6. 最后再定义第一个 CMS 扩展阶段

## 13. 新需求判断规则

面对任何新需求时，先回答一个问题：

它是在提升当前个人博客体验，还是本质上属于未来 CMS 阶段？

如果它提升的是当前个人博客体验，就可以进入本计划。

如果它主要服务于管理、鉴权、运营后台或编辑工作台，则应放入未来阶段的 specs/plans 中处理。

## 14. 当前阶段完成定义

当前阶段满足以下条件时，可以认为基本完成：

- 公开博客整体足够完整、连贯、成熟
- MDX 工作流稳定且易用
- 搜索与 metadata 可靠
- 项目易于维护与部署
- 当前阶段没有任何“必须依赖 CMS 才能成立”的关键缺口

## 15. 文档权威说明

在当前个人博客阶段：

- `docs/PRD.md` 和本中文文档共同用于表达当前产品目标
- `docs/development-plan.md` 和本中文文档共同用于表达当前实施计划

未来 CMS 工作不应反向重写当前阶段，而应通过 `docs/superpowers/` 下的新分阶段文档独立定义。
