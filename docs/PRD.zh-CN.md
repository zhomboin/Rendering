# Rendering 产品需求文档

日期：2026-03-19
项目：Rendering
文档状态：Active
文档角色：当前个人博客阶段的唯一中文 PRD

## 1. 产品定义

`Rendering` 在当前阶段是一个个人技术博客，而不是 CMS 产品。

在这一阶段，产品的核心是一套面向公开访问的发布型站点，用于承载关于前端系统、渲染、界面行为、动效、性能以及工程思考的长文写作。它应该体现明确的作者表达、稳定的内容结构，以及高质量的阅读体验。

当前阶段优先关注：

- 优秀的公开阅读体验
- 清晰的个人品牌和写作气质
- 稳定的本地内容工作流
- 面向已发布文章的搜索与 SEO
- 即使没有后台管理能力也显得完整的博客前台

这一阶段不以多用户发布平台、登录系统或内容管理后台为目标。

## 2. 产品愿景

`Rendering` 的长期愿景，是逐步成长为一个以写作为核心的技术出版产品，并在未来具备更丰富的发布与管理能力。但当前里程碑更聚焦：

- 先做成一个高质量的个人博客
- 先验证视觉语言、内容模型和阅读体验
- 在不引入多余复杂度的前提下，为未来扩展保留架构空间

这个博客应该像一个公开的技术思考工作台，而不是一个通用模板站点，或一个过早预埋后台逻辑的 CMS 外壳。

## 3. 问题陈述

很多个人工程博客常见的失败方式主要有两种：

- 视觉和结构过于普通，缺乏鲜明身份
- 过早投入后台与平台能力，反而拖慢了真正重要的阅读体验建设

`Rendering` 要解决的正是这两个问题：先交付一个具有辨识度、阅读舒适、内容导向明确的个人博客，再在未来视需要扩展后台管理能力。

## 4. 当前阶段目标

交付一个完整的个人博客体验，至少支持：

- 使用本地 MDX 文件发布技术文章
- 通过首页、归档页、标签页和文章页浏览内容
- 通过静态全文搜索发现内容
- 为公开内容提供基础 SEO 元数据
- 建立一套围绕冷峻终端 cyber 风格的稳定视觉系统

当前阶段的成功标准，是这个站点在没有登录、后台面板、数据库工作流的情况下，也已经是一个成立且完整的公开博客产品。

## 5. 目标用户

核心使用者：

- 博客作者本人，用来发布个人技术写作

核心读者：

- 前端工程师
- Web 性能相关从业者
- 关注设计工程交叉领域的读者
- 对渲染、UX 系统和技术写作感兴趣的开发者

次级受众：

- 通过博客判断作者思考能力、品味与表达能力的潜在合作方、招聘方与同行

## 6. 产品原则

### 6.1 写作为先

产品的存在是为了服务技术写作与阅读，装饰性特性不能压过内容本身。

### 6.2 强识别度，但受控表达

视觉方向应当鲜明、有记忆点，但不能牺牲可读性。整体气质应当是技术化、冷静、克制，而不是浮夸或过载。

### 6.3 默认静态友好

当前系统应优先采用简单、可靠、适合静态部署的架构，让内容编写、构建与发布保持直接清晰。

### 6.4 为未来预留，而不是为未来膨胀

当前架构需要为未来的登录和 CMS 扩展保留空间，但不能因此把当前阶段提前做成一个“伪 CMS”。

## 7. 当前阶段范围内功能

### 7.1 公开博客体验

- 首页
- 博客归档页
- 博客详情页
- 标签页
- About 页面

### 7.2 内容系统

- 位于 `content/posts` 的本地 MDX 内容
- 由 frontmatter 驱动的元数据
- 草稿过滤
- 标签聚合
- 上一篇 / 下一篇文章导航

### 7.3 搜索与发现

- 基于 Pagefind 的全站静态搜索
- 键盘快捷键与搜索弹层 UI
- 可见的标签 / 主题导航

### 7.4 SEO 基础能力

- 路由级 metadata
- 基于 frontmatter 的文章 metadata
- canonical URL
- 基础 Open Graph 信息
- 基础 Twitter 信息

### 7.5 视觉与交互系统

- 冷峻终端 cyber 风格视觉语言
- 适合长文阅读的文章表面设计
- 克制的动效
- 清晰的 focus 状态
- 适配移动端与桌面端的响应式行为

## 8. 当前阶段明确不做的内容

以下能力不属于当前个人博客阶段：

- 登录
- 作者鉴权
- 管理后台
- 在线编辑器
- 媒体资源库
- 角色与权限系统
- 基于数据库的文章管理
- 定时发布工作流
- 评论审核后台
- 数据分析后台
- 多作者发布工作流

这些能力未来可以成为有效阶段目标，但并不是当前产品成功所必须具备的条件。

## 9. 信息架构

当前阶段的公开信息架构为：

- `/` 首页
- `/blog` 归档页
- `/blog/[slug]` 文章详情页
- `/tags` 主题标签页
- `/about` 作者与项目说明页

全局导航应保持简洁、偏编辑型：

- Home
- Blog
- Tags
- About
- Search 入口

## 10. 内容模型

每篇文章应支持以下内容契约：

- `title`
- `description`
- `publishedAt`
- `tags`
- `draft`
- `coverImage` 可选，留待未来扩展

规则：

- 文件名默认作为 slug
- 草稿不会进入公开路由
- `description` 同时作为摘要与 meta description 的来源
- tags 同时服务于内容发现和信息架构

## 11. 用户体验要求

### 11.1 首页

首页应承担品牌建立、编辑气质表达、精选内容展示和主题入口作用，而不是传统营销型 landing page。

### 11.2 归档页

归档页应支持高密度、快速扫描式的内容浏览，具备清晰元数据和低摩擦导航体验。

### 11.3 文章阅读页

文章页是整个产品中优先级最高的体验，必须重点保证：

- 阅读舒适度
- 层级清晰
- 代码块与长文排版质量
- 低干扰、低噪声的动效表达

### 11.4 搜索

搜索应该像编辑型内容产品的自然延伸，而不是一个通用应用启动器。

## 12. 视觉方向

当前选定的视觉方向保持不变：

- 冷峻终端 cyber 风格

关键视觉特征：

- 深碳黑与石墨表面
- 青蓝与冰蓝强调色
- 品红仅用于极少量关键高亮
- 面板式构图
- 细密网格和扫描线暗纹
- 受控的 glow 与 motion

应避免的反模式：

- 泛紫色 AI 渐变
- 过强 bloom 和 blur
- 用仪表盘式 chrome 压过正文内容
- 与阅读竞争注意力的装饰性动画

## 13. 技术方向

当前阶段的技术方向应支持个人博客目标，并避免引入不必要复杂度：

- Next.js App Router
- React
- TypeScript
- 本地 MDX 内容
- 使用 `gray-matter` 解析 frontmatter
- 使用 `@mdx-js/mdx` 编译 MDX
- 使用 `remark-gfm` 支持 markdown 扩展能力
- 使用 Pagefind 提供静态搜索

推荐的整体架构为：

- 基于本地文件的内容仓库层
- 静态友好的路由方式
- 基于路由的 metadata 生成
- 可复用 UI 组件
- 当前阶段不依赖数据库

## 14. 成功标准

当前阶段满足以下条件时，即可认为成功：

- 站点在没有后台管理能力的情况下，依然作为个人博客成立
- 通过本地 MDX 发布新文章足够直接
- 所有公开页面视觉上一致、完整
- 文章页提供可信的长文阅读体验
- 搜索可以有效帮助读者发现内容
- 站点具备扎实的 SEO 基础

## 15. 风险

### 15.1 目标漂移到 CMS

风险：
未来的平台化想法可能模糊当前阶段边界，导致过早复杂化。

缓解方式：
以本 PRD 作为当前范围边界，把 CMS 能力视为后续独立阶段。

### 15.2 风格压过内容

风险：
cyber-terminal 风格如果控制不好，会伤害阅读体验。

缓解方式：
让文章页比品牌页更克制，把排版和间距优先级放在装饰效果之前。

### 15.3 内容工作流脆弱化

风险：
过早引入过多工具，会让写作与发布流程更慢而不是更好。

缓解方式：
当前阶段保持本地 MDX 作为权威写作流程。

## 16. 未来扩展路径

当个人博客阶段稳定后，`Rendering` 可以通过 `docs/superpowers/specs/` 与 `docs/superpowers/plans/` 中的分阶段文档继续扩展。

未来可能的扩展阶段包括：

- 作者登录
- 私有后台界面
- 文章管理 UI
- 草稿与发布工作流
- 基于数据库的内容存储
- 资源管理
- 评论与读者互动系统

这些能力应通过独立阶段的规格文档与实现计划来描述，而不应反向混入当前个人博客阶段。

## 17. 文档策略

在当前阶段，这份文件是唯一的中文 PRD。

以下目录保留给未来多阶段扩展使用：

- `docs/superpowers/specs/`
- `docs/superpowers/plans/`

这些文档应当服务于未来 CMS 或平台阶段，而不是重新定义当前个人博客的目标范围。
