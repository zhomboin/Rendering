# 内容迁移指南

日期：2026-03-21
适用项目：Rendering
文档角色：精选原创内容迁移工作流

## 目标

这份指南用于把用户在 CSDN 上的原创技术文章，迁移成适合 `Rendering` 当前风格的本站原生文章。

迁移原则不是“复制”，而是：

- 保留技术事实
- 重写结构与节奏
- 统一标题、摘要、标签和结尾口径
- 去掉平台痕迹，让文章看起来像本来就写在本站

## 当前迁移边界

- 本站视为首发站
- 后续由用户在 CSDN 回链到本站
- 当前只做人工精选迁移，不做批量导入
- 当前首批选题聚焦：数据库 / 系统 / 服务器

## 首批推荐迁移文章

1. `MySQL MVCC 详解`
2. `数据库AICD特性之--隔离性 Isolation`
3. `Redis 哨兵模式部署--docker版本`
4. `ubuntu安装配置fail2ban`
5. `Frp搭建`（可选第五篇）

## 单篇迁移流程

### 1. 先判断是否值得迁入

优先选择：

- 主题足够扎实
- 不是纯平台公告或活动文
- 适合形成数据库 / 系统 / 服务器的主线
- 具备长期可搜索价值

### 2. 重写标题

标题要更像本站原生文章，而不是平台发文标题。

建议原则：

- 更明确问题对象
- 避免口语化冗余
- 尽量让技术名词出现在标题里
- 保留搜索价值，不追求夸张标题

### 3. 重写摘要

摘要不是重复标题，而是给出：

- 文章解决的核心问题
- 读者能学到什么
- 适用场景或边界

### 4. 调整结构

建议统一成以下节奏：

1. Why this matters
2. What is happening
3. Walkthrough
4. Practical checks
5. Closing notes

如果原文结构较好，可以保留大部分技术顺序，但要把平台式碎片段落整理成更连贯的小节。

### 5. 清理平台痕迹

迁移时应删除：

- CSDN 平台口吻
- 活动文案
- 自动生成式结尾
- 平台特有的目录提示或引导语
- 与本站无关的外链说明

### 6. 统一代码块与命令行格式

迁移时检查：

- 命令是否可复制
- 代码缩进是否统一
- shell、SQL、Java 等语言是否清晰分段
- 是否需要把长代码拆成更小的可讲解片段

### 7. 选择标签

当前优先围绕：

- `database`
- `systems`
- `servers`
- `linux`
- `redis`
- `mysql`

避免每篇文章都制造新标签，优先复用能形成归档厚度的标签。

### 8. 判断是否需要图片或图示

只有在图片能明显提升理解时才添加：

- 架构图
- 主从关系图
- 请求流向图
- 配置前后对比图

不要为了装饰而补图。

### 9. 最终发布前检查

每篇迁移文章发布前都应确认：

- frontmatter 完整
- 标题、摘要、标签已重写
- 正文段落顺序清晰
- 代码块可读
- 结尾不是平台模板话术
- 构建和内容校验通过

## 建议执行顺序

1. 先创建草稿文件
2. 复制技术核心内容
3. 重写标题与摘要
4. 调整结构与节奏
5. 统一标签和图文形式
6. 本地验证
7. 再决定是否进入下一篇
